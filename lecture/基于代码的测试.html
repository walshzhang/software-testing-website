# 4.1 控制流图

  • 基本块：一个顺序的语句序列，只有一个入口点和一个出口点

  • 基本块例子

    • 为什么 1 号基本块到第第 5 行结束？
    • 为什么第 10 行 while 语句构成一个基本块？
    • 第 1，17，7，13 行被忽略
    • 为什么第 16 行单独构成基本块？

  • 控制流图：描述程序中的控制流，也称流图。G=(N,E)G=(N,E)G=(N,E)，NNN 表示图中节点的集合，每个结点对应一个基本块。EEE 表示边的集合，每条边表示基本块之间的控制流。边 (i,j)(i,j)(i,j) 表示从基本块 NiN_iN​i​​ 到 NjN_jN​j​​ 的控制流

  • 流图中没有入边的结点称为起始结点，没有出边的结点称为终止结点

  • 控制流图例子

  • 考虑流图 G=(N,E)G=(N,E)G=(N,E)。假设 np,nq,nr,nsn_p,n_q,n_r,n_sn​p​​,n​q​​,n​r​​,n​s​​ 是 NNN 中的结点，对任意 0<i<k0<i<k0<i<k，如果有 ei=(np,nq)e_i=(n_p,n_q )e​i​​=(n​p​​,n​q​​) 且 e(i+1)=(nr,ns)e_(i+1)=(n_r,n_s)e​(​​i+1)=(n​r​​,n​s​​)，则 nq=nrn_q=n_rn​q​​=n​r​​，则此 kkk 条边序列 (e_1,e_2,⋯,e_k) 是一条长度为 kkk 的路径

  • 对于任何 n,m∈N，如果存在一条从 nnn 到 mmm 的路径，则称 mmm 是 nnn 的后继，或称 nnn 是 mmm 前驱。若 m≠nm \neq nm≠n，则 mmm 是 nnn 的真后继，nnn 是 mmm 的真前驱。如果存在 (n,m)∈E(n,m) \in E(n,m)∈E，则称 mmm 是 nnn 的直接后继，nnn 是 mmm 的直接前驱。结点 nnn 的直接前驱和直接后继分别记为 prev(n)prev(n)prev(n) 和 succ(n)succ(n)succ(n)。起始结点没有前驱，终止结点没有后继。如果存在循环，结点可为自身的前驱和后继

  • 流图中的一条路径，如果其首结点为起始结点，末结点为终止结点，则称该路径是完整的。

  • 流图中的一条路径 ppp，如果至少存在一条测试用例，其在执行程序时能够遍历 ppp，则称 ppp 是可达的，否则称其是不可达的

# 4.2 测试充分性

  • 假设软件 PPP 要满足功能需求集合 RRR，记为 (P,R)(P,R)(P,R)。RRR 中包含 nnn 个需求，记为 R_1,R_2,⋯,R_n。再假设测试集 TTT 包含了 kkk 个测试用例用于确认 PPP 是否满足 RRR 中的所有需求，并假设 TTT 中的每个用例都执行成功

  • "充分性“用来度量一个给定的测试集是否能验证软件 PPP 满足其需求。当测试集满足测试准则 CCC 时，即认为其相对于 CCC 是充分的

  • 例1：考虑编写程序 PPP，其需求 RRR 如下：

    • R1R_1R​1​​: 从键盘输入两个整数 xxx 和 yyy
    • R2.1R_{2.1}R​2.1​​: 当 x<yx<yx<y 时，求 xxx 和 yyy 之和，并在屏幕上输出
    • R2.2R_{2.2}R​2.2​​: 当 x≥yx \geq yx≥y 时，求 xxx 和 yyy 之积，并在屏幕上输出

  • 需求覆盖：如果针对 RRR 中的每个需求 rrr，测试集 TTT 中至少有一个用例能够证明 PPP 满足 rrr，则认为 TTT 对于 (P,R)(P,R)(P,R) 是充分的

  • 覆盖域：用于度量测试集充分性的有限集。每个测试准则都可导出一个覆盖域

    • 假设要度量测试集 TTT 的充分性。给定覆盖域 CeC_eC​e​​，它有 n≥0 个元素。如果 CeC_eC​e​​ 中的每个元素 eee，在 TTT 中都至少有一个测试用例能够测试它，则称 TTT 覆盖 eee。如果 TTT 能覆盖 CeC_eC​e​​ 中的所有元素，称 TTT 对于 CeC_eC​e​​ 是充分的。如果 TTT 只能覆盖 CeC_eC​e​​ 中的 k≤nk \leq nk≤n 个元素，称 TTT 对于 CeC_eC​e​​ 是不充分的。比率 k/nk/nk/n 代表 TTT 对于 CCC 的充分度，也称 TTT 对于 C,R,PC,R,PC,R,P 的覆盖率

  • 例2：考虑例 1 中的 P,R,CP,R,CP,R,C，令 T={⟨x=3,y=4⟩},Ce={R1,R2.1,R2.2}T = \{ \langle x = 3, y = 4 \rangle \}, C_e=\{R_1,R_{2.1},R_{2.2}\}T={⟨x=3,y=4⟩},C​e​​={R​1​​,R​2.1​​,R​2.2​​}. 此时 TTT 覆盖 R1R_1R​1​​ 和 R2.1R_{2.1}R​2.1​​，但没覆盖 R2.2R_{2.2}R​2.2​​。因此 TTT 对于 CeC_eC​e​​ 是不充分的，覆盖率为 2/32 / 32/3

  • 例3：例 1 中的 PPP 所对应的代码有两条路径 p1p_1p​1​​ 和 p2p_2p​2​​，分别对应条件 x<yx<yx<y 和 x≥yx \geq yx≥y. TTT 对于路径覆盖准则不充分，其只能覆盖 p1p_1p​1​​，因此覆盖率为 0.50.50.5

  • **路径覆盖准则：**如果程序 PPP 中的每条路径都被遍历至少一次，则认为测试集 TTT 针对 (P,R)(P,R)(P,R) 是充分的

    • 条件语句会引起路径数量的指数级增长
    • 循环的存在将极大地增加路径的数量，每循环一次，路径数量至少增加 1

# 4.3 测试增强

  • 虽然测试集对于某些测试准则是充分的，但不能保证程序没有错误。不充分的测试集意味着测试不足。此时，如果能提高测试集的覆盖率，就能提高发现错误的几率，即测试增强

  • 例4：例 3 中的 TTT 对于 CCC 的覆盖率只有 0.50.50.5，如果往 TTT 中添加测试用例 ⟨x=3,y=1⟩\langle x=3, y=1 \rangle⟨x=3,y=1⟩，则扩充后的测试集 T′T^\primeT​′​​ 能够覆盖 {p1,p2}\{p_1,p_2\}{p​1​​,p​2​​}，覆盖率为 1

  • 通过度量充分性来增强测试

    • 如果测试集 TTT 对于准则 CCC 是充分的，则无需增强，退出。否则进入下一步
    • 对每个未被覆盖的 e∈C_e，执行以下步骤
      • 构造测试用例 ttt，它能够覆盖 eee，把 ttt 加入到 TTT 中
      • 针对 ttt 执行 PPP，如果执行不正确，则修改 PPP，重复此步骤直到 PPP 执行正确

  • 例5：PPP 为计算 xyx^yx​y​​ 的程序。xxx 和 yyy 为整数。若 y<0y<0y<0，输出错误信息

    • 测试准则 CCC：如果测试集 TTT 对 xxx 和 yyy 中的每一个，至少经过一次等于 000 和不等于 000，则 TTT 是充分的 Ce={x=0,y=0,x≠0,y≠0}C_e = \{ x = 0, y = 0, x \neq 0,y \neq 0 \}C​e​​={x=0,y=0,x≠0,y≠0}
    • T={⟨x=0,y=1⟩,⟨x=1,y=0⟩}T=\{\langle x=0,y=1 \rangle, \langle x=1,y=0 \rangle \}T={⟨x=0,y=1⟩,⟨x=1,y=0⟩} 相对于 CCC 是充分的

  • 例6：将例 5 中的测试准则替换为路径覆盖准则。由于 whilewhilewhile 语句的存在，流图中路径的个数具有不确定性，其依赖 yyy 的值。因此其路径个数可能会非常大，此时，无法确定路径覆盖准则的覆盖域。此时，可将路径覆盖准则简化为 C′C^\primeC​′​​：

    • C′C^\primeC​′​​: 如果测试集 TTT 测试了所有路径，则它是充分的。若程序包含循环，则只要 TTT 遍历过循环体 000 次和 111 次即可
    • 根据 C′C^\primeC​′​​，可导出 Ce′={p1,p2,p3}C_e^\prime=\{p_1,p_2,p_3\}C​e​′​​={p​1​​,p​2​​,p​3​​}。其中，
      • p1=(1,2,3,4,5,7,9)p_1=(1,2,3,4,5,7,9)p​1​​=(1,2,3,4,5,7,9)
      • p2=(1,2,3,4,5,6,5,7,9)p_2=(1,2,3,4,5,6,5,7,9)p​2​​=(1,2,3,4,5,6,5,7,9)
      • p3=(1,2,3,8,9)p_3=(1,2,3,8,9)p​3​​=(1,2,3,8,9)
    • 测试增强：T′=T∪{⟨x=5,y=−1⟩}T^\prime=T \cup \{\langle x=5,y=-1 \rangle \}T​′​​=T∪{⟨x=5,y=−1⟩}

    • 对于复合条件 CCC 中的一个简单条件，如果不能证明它对 CCC 的输出结果有独立影响，则它是无效的简单条件

  • 修正条件/判定覆盖MC/DC（Modified Condition/Decision Coverage）：满足需求 RRR 的软件 PPP 的测试集 TTT 针对 MC/DC 覆盖准则是充分的，如果对 PPP 执行 TTT 中的测试用例，满足如下要求：

    • 条件覆盖：PPP 中的每一个简单条件都取过 true 和 false
    • 判定覆盖：PPP 中的每一个判定都得出过所有可能的输出结果
    • MC覆盖：PPP 中的复合条件 CCC 中的每个简单条件对 CCC 的输出结果的影响是独立的

  • 如果判定中的简单条件个数为 kkk，多重条件覆盖要求组合数量为 2k2^k2​k​​，MC/DC要求的组合可降为 k+1k + 1k+1 个，两者具有几乎相同的测试效果。

    • 当 k=32k = 32k=32 时，测试用例个数降为 33 个，执行时间从 24.75 天缩短为 33 毫秒
    • 民航 DO-178C《机载系统合格审定过程中的软件考虑》明确指出：A级软件必须进行MC/DC测试
    • ISO 26262-6 《道路车辆功能安全》要求：A级软件必须进行MC/DC测试

# 4.5 测试准则包含关系

  • 包含性：给定测试集 TTT，它针对测试准则 C1C_1C​1​​ 是充分的，针对另一个测试准则 C2C_2C​2​​ 是否充分？
  • 路径覆盖 ⇒\Rightarrow⇒ 多重条件覆盖 ⇒\Rightarrow⇒ MC/DC ⇒\Rightarrow⇒ 条件/判定覆盖 ⇒\Rightarrow⇒ 判定覆盖 ⇒\Rightarrow⇒ 语句覆盖

  • 条件覆盖 ⇏\nRightarrow⇏ 判定覆盖

public class CountWords {
    // 统计给定字符串中以 s 或 r 结尾的单词的个数
    // 当出现非字母字符时，视为单词结束
    public int count(String str) {
        int words = 0;
        char last = ' ';
        int i = 0;
        
        while ( i < str.length()) {
            if (!isLetter(str.charAt(i)) && (last == 's' || last == 'r')) {
                words++;
            }
            
            last = str.charAt(i); 
            i++;
        }
        
        if (last == 'r' || last == 's') { 
            words++;
        }
        
        return words;
    }
}
